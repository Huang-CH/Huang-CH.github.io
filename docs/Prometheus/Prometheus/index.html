    title: Prometheus

    Prometheus简介 #

    官方网站：https://prometheus.io

    Prometheus 是一个开源系统，最初是在 SoundCloud 上构建的监控和警报工具包。自2012年成立以来，许多公司和组织都采用了Prometheus，该项目具有非常活跃的开发人员和用户 社区。它现在是一个独立的开源项目，并独立于任何公司进行维护。强调这一点，并澄清该项目的治理结构，Prometheus于2016年加入 云原生计算基金会 作为继 Kubernetes 之后的第二个托管项目。

    Prometheus 将所有采集到的样本（sample）以时间序列的方式保存在时序数据库中。

    样本（sample）由指标（metric）、时间戳（timestamp）、样本值（value）组成。

    指标（metric）包含四种类型：

    官网链接：https://prometheus.io/docs/concepts/metric_types/#metric-types

    • Counter：Counter（计数器）是一个累积型的指标，其值只能增加或重置为零。例如，可以使用Counter来记录开机时间、CPU使用时间、网络请求总数量。特点是描述“只增不减”的数据类型。

      注意：不要使用Counter来暴露可以减少的值。例如，不要对当前正在运行的进程数量使用Counter；而是使用Gauge（仪表）。

      计数器的客户端库使用文档：

      • Go
      • Java
      • Python
      • Ruby
      • .Net

    • Gauge：Gauge（传感器）是一个可以表示任意上下起伏数值的指标，其返回值是一个瞬时状态。例如：温度、当前内存可用容量、硬盘可用容量。特点是描述“可增可减”的数据类型。

      Gauge的客户端库使用文档：

      • Go
      • Java
      • Python
      • Ruby
      • .Net

    • Histogram：Histogram（直方图）是一个统计分析数据分布情况的指标。该指标对观测结果进行采样（通常如请求持续时间或响应大小），并在可配置的buckets中计数。它还提供了所有观察到的值的总和。该指标的对于不同区间样本个数的计算是在服务端完成的。

      Histogram的基本度量名称为<basename>，在刮擦期间暴露了多个时间序列：

      观察buckets的累积计数器，暴露为<basename>_bucket{le="<upper inclusive bound>"}

      所有观察到的值的总和，公开为<basename>_sum

      已观察到的事件计数，暴露为<basename>_count（与上面的<basename>_bucket{le="+Inf"}相同）

      使用 histogram_quantile() 函数从直方图甚至直方图聚合中计算分位数。直方图也适合计算Apdex分数。在存储桶上操作时，请记住直方图是累积的。有关直方图使用情况和摘要差异的详细信息，请参阅直方图和摘要。

      Histograms的客户端库使用文档:

      • Go
      • Java
      • Python
      • Ruby
      • .Net

    • Summary：Summary（摘要）也是一个统计分析数据分布情况的指标。但该指标对于不同区间内样本个数的计算是在客户端完成的。

      具有<basename>基本度量名称的摘要在抓取期间暴露了多个时间序列：

      流式φ-quantiles（0 ≤ φ ≤ 1）观察到的事件，暴露为<basename>{quantile="<φ>"}

      所有观察到的值的总和，公开为<basename>_sum

      已观察到的事件计数，暴露为<basename>_count

      有关φ分位数、摘要用法和与直方图的差异的详细解释，请参阅直方图和摘要。

      客户端库使用文档摘要：

      • Go
      • Java
      • Python
      • Ruby
      • .Net

    Prometheus常用函数 #

    官网链接：https://prometheus.io/docs/prometheus/latest/querying/functions/

    rate函数 #

    rate()函数是专门搭配Counter（计数器）类型数据来使用的。它的功能是计算出一个时间段内平均每秒的增量。

    示例：

    在Prometheus中获取1分钟内每秒网络的接收字节数

    rate(node_network_receive_bytes_total[1m])

    increase函数 #

    increase()函数也是专门搭配Counter（计数器）类型数据来使用的。它的功能是计算出一段时间内增加的总量。

    示例：

    在Prometheus中获取1分钟内网络的接收字节数

    increase(node_network_receive_bytes_total[1m])

    经验：在遇到Counter（计数器）类型数据的时候首先要做的就是套上rate()函数或者instance()函数。

    sum函数 #

    sum()函数会把所有监控节点结果集进行求和，如果需要分组的话，则需要通过by()来实现。

    示例：

    对所有监控节点的网络接收字节数束进行求和

    sum(rate(node_network_receive_bytes_total[1m]))

    对所有监控节点的网络接收字节数进行分组求和

    sum(rate(node_network_receive_bytes_total[1m])) by (instance)

    topk函数 #

    topk()函数可以实现取前几位最高值的功能。

    示例：

    取所有监控节点中网络接收字节数前3的节点

    topk(3,rate(node_network_receive_bytes_total[1m]))

    topk()函数在实际应用中一般不会用来观测数据，而是用来进行瞬时报警。

    count函数 #

    count()函数用于计算出符合条件的节点数量。

    示例：

    count(rate(node_cpu_seconds_total[1m]) > 0.5)

    count()函数一般用于监控判断，例如，一个集群中有100台节点，当CPU使用率达到80%的节点数量达到50台的时候，就需要触发报警。

    Prometheus架构 #

    安装Prometheus #

    创建软件安装目录

    mkdir -p /opt/installed

    下载 Prometheus 2.45.0 长期支持版

    wget https://github.com/prometheus/prometheus/releases/download/v2.45.0/prometheus-2.45.0.linux-amd64.tar.gz -P /opt/installed

    解压缩安装包

    tar -xvf /opt/installed/prometheus-2.45.0.linux-amd64.tar.gz -C /opt/

    创建软连接

    ln -s /opt/prometheus-2.45.0.linux-amd64 /usr/local/bin/prometheus

    创建 Prometheus 数据目录

    mkdir /data/prometheus

    创建 Prometheus 用户（可跳过）

    useradd -M -s /usr/sbin/nologin prometheus

    将 /opt/ 目录授权给 Prometheus 用户（可跳过）

    chown -R prometheus:prometheus /opt/prometheus-2.45.0.linux-amd64

    进入 Prometheus 目录，执行 Prometheus 程序就可以直接启动 Prometheus（生产环境不推荐）

    nohup /opt/prometheus-2.45.0.linux-amd64/prometheus --config.file=/opt/prometheus-2.45.0.linux-amd64/prometheus.yml >/dev/null 2>&1 &

    为 Prometheus 创建 systemd 服务（也可以在 /usr/lib/systemd/system/ 目录中创建）

    cat > /etc/systemd/system/prometheus.service <<EOF
[Unit]
Description=Prometheus Server
Documentation=https://prometheus.io/docs/intorduction/overview/
After=network-online.target

[Service]
Type=simple
# User=prometheus
# Group=prometheus
Restart=on-failure
ExecStart=/opt/prometheus/prometheus \
  --config.file=/opt/prometheus/prometheus.yml \
  --storage.tsdb.path=/data/prometheus \
  --storage.tsdb.retention.time=365d \
  --web.enable-lifecycle
    
[Install]
WantedBy=multi-user.target
EOF

    常用启动参数介绍

    --config.file="prometheus.yml"：指定配置文件，默认在安装目录下。

    --web.read-timeout=5m：请求链接的最大等待时间，默认5分钟。

    --web.max-connections=512：最大同时链接数，默认512。

    --storage.tsdb.path="/data"：指定采集数据的存储路径，默认存储在/data。

    --storage.tsdb.retention.time=30d：指定采样在存储中的保留时间，一般设置为15至30天为宜。

    --query.timeout=2m：指定查询超时时间，默认2分钟。

    --query.max-concurrency=20：指定查询的最大并发数，默认20个。

    重启载入配置文件

    systemctl daemon-reload

    启动 Prometheus 服务并配置 Prometheus 随系统启动

    systemctl start prometheus.service && systemctl enable prometheus.service

    Prometheus配置文件说明 #

    Prometheus配置文件路径： /opt/prometheus-2.45.0.linux-amd64/prometheus.yml

    一级配置项 # 

    ＃全局配置（如果有内部单独设定，会覆盖这个参数）
global:

＃告警插件定义。这里会设定alertmanager这个报警插件。
alerting:

＃告警规则。 按照设定参数进行扫描加载，用于自定义报警规则，其报警媒介和route路由由alertmanager插件实现。
rule_files:

＃采集配置。配置数据源，包含分组job_name以及具体target。又分为静态配置和服务发现
scrape_configs:

＃用于远程存储写配置
remote_write:

＃用于远程读配置
remote_read:

    二级配置项 # 

    # my global config
global:
  scrape_interval: 15s   # 全局默认的数据拉取间隔
  evaluation_interval: 15s   # 全局默认的规则(主要是报警规则)拉取间隔
 
# Alertmanager configuration
alerting:
  alertmanagers:
  - static_configs:
    - targets:
      # - alertmanager:9093
 
# Load rules once and periodically evaluate them according to the global 'evaluation_interval'.
rule_files:
  # - "first_rules.yml"
  # - "second_rules.yml"
 
# A scrape configuration containing exactly one endpoint to scrape:
# Here it's Prometheus itself.
scrape_configs:
  # The job name is added as a label `job=` to any timeseries scraped from this config.
  - job_name: 'Prometheus'
 
    # metrics_path defaults to '/metrics'
    # scheme defaults to 'http'.
 
    static_configs:
    - targets: ['localhost:9090']
      labels:
        instance: 'Prometheus服务器'
    
  # 配置多个任务（job）
  - job_name: 'Node_Exporter'
    static_configs:
    - targets: ['10.10.10.100:9100']
    # - targets: ['10.10.10.10:8080','10.10.10.11:8080','10.10.10.12:8080']   # 集群配置
      labels:
        instance: 'Node_Exporter'
    - targets: ['10.10.10.10:9100']
      labels:
        instance: '负载均衡服务器'
    - targets: ['10.10.10.11:9100']
      labels:
        instance: 'Nginx-01服务器'
    - targets: ['10.10.10.12:9100']
      labels:
        instance: 'Nginx-02服务器'

    安装Node_Exporter #

    Prometheus exporter for hardware and OS metrics exposed by *NIX kernels, written in Go with pluggable metric collectors.

    The Windows exporter is recommended for Windows users. To expose NVIDIA GPU metrics, prometheus-dcgmcan be used.

    Prometheus exporter是一款使用Go语言编写的指标收集器，用于收集Unix、Linux内核的硬件和系统指标。

    Windows用户建议使用Windows环境下的exporter。使用 prometheus-dcgm可以暴露NVIDIA GPU的指标。

    Node_Exporter安装在需要监控的服务器上。

    useradd -M -s /usr/sbin/nologin prometheus

    将/opt/Node_Exporter目录授权给 Prometheus 用户（可跳过）

    chown -R prometheus:prometheus /opt/node_exporter-1.6.1.linux-amd64

    为 Node_Exporter 创建 systemd 服务

    将 Blackbox_Exporter 添加到 Prometheus，在 /opt/prometheus-2.45.0.linux-amd64/prometheus.yml 文件中添加如下内容：

    scrape_configs:
  - job_name: 'blackbox'
    metrics_path: /probe
    params:
      module: [http_2xx]  # Look for a HTTP 200 response.
    static_configs:
      - targets:
        - http://prometheus.io    # Target to probe with http.
        - https://prometheus.io   # Target to probe with https.
        - http://example.com:8080 # Target to probe with http on port 8080.
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: 127.0.0.1:9115  # The blackbox exporter's real hostname:port.

    HTTP probes can accept an additional hostname parameter that will set Host header and TLS SNI. This can be especially useful with dns_sd_config:

    scrape_configs:
  - job_name: blackbox_all
    metrics_path: /probe
    params:
      module: [ http_2xx ]  # Look for a HTTP 200 response.
    dns_sd_configs:
      - names:
          - example.com
          - prometheus.io
        type: A
        port: 443
    relabel_configs:
      - source_labels: [__address__]
        target_label: __param_target
        replacement: https://$1/  # Make probe URL be like https://1.2.3.4:443/
      - source_labels: [__param_target]
        target_label: instance
      - target_label: __address__
        replacement: 127.0.0.1:9115  # The blackbox exporter's real hostname:port.
      - source_labels: [__meta_dns_name]
        target_label: __param_hostname  # Make domain name become 'Host' header for probe requests
      - source_labels: [__meta_dns_name]
        target_label: vhost  # and store it in 'vhost' label

    安装Alertmanager #

    The Alertmanager handles alerts sent by client applications such as the Prometheus server. It takes care of deduplicating, grouping, and routing them to the correct receiver integrations such as email, PagerDuty, OpsGenie, or many other mechanisms thanks to the webhook receiver. It also takes care of silencing and inhibition of alerts.

    Alertmanager处理Prometheus服务器发送的报警。Alertmanager内部的webhook receiver可是实现数据去重和分组，并将它们路由到正确的集成接收器（如：电子邮件、PagerDuty、OpsGenie或其他机制），同时，他还负责报警的沉默与抑制。

    Alertmanager可以安装在Prometheus服务器上，也可以安装在独立的服务器上。

    将 /opt/Alertmanager 目录授权给 Prometheus 用户（可跳过）

    如果配置了热加载，也可以使用如下命令：

    curl -x POST http://localhost:9090/-/reload

    向 Alertmanager 告警配置文件中添加触发器，在 /opt/alertmanager-0.26.0.linux-amd64/alert.yml 文件添加如下内容：

    cat > /opt/alertmanager-0.26.0.linux-amd64/alert.yml <<EOF
groups:
- name: Prometheus Alert
  rules:
  # 设置告警
  expr: up == 0
  for: 60s
  labels:
    severity: critical
  annotations:
    instance: "{{ $labels.instance }}"

    systemctl restart alertmanager

    Prometheus数据采集方式 #

    Pull方式采集数据（主动） #

    Pull方式采集数据是指先在客户端（被监控机）上安装各类 exporters ，并以守护进程的方式运行。 exporters 本身也是一个HTTP_Server，可以对HTTP请求作出响应和返回请求数据。Prometheus使用Pull方式采集数据，其实就是通过HTTP请求的方式去获取每个节点上 exporters 所采集的数据。

    Push方式采集数据（被动） #

    Push方式采集数据是指在任意节点（可以是服务端、客户端或独立节点）安装 PushGateway 插件，通过自己开发的各种监控脚本把监控数据组织成 Key / Value 的形式发送给 PushGateway，然后再由 PushGateway 发送给Prometheus服务。

    常用监控指标的计算公式 #

    系统信息 #

    获取系统信息

    node_uname_info{job="node_exporter"}

    系统运行时间

    sum(time() - node_boot_time_seconds{job="node_exporter"}) by (instance)

    time() ：系统当前时间

    node_boot_time_seconds：系统启动时间

    CPU #

    CPU核数

    count(node_cpu_seconds_total{job="node_exporter",mode='system'}) by (instance)

    CPU使用率

    (sum by (instance) (increase(node_cpu_seconds_total{mode!="idle"}[1m])) / (sum  by (instance)(increase(node_cpu_seconds_total[1m])) ) * 100

    CPU IO Wait率

    sum(rate(node_cpu_seconds_total{job="node_exporter",mode="iowait"}[1m])) / sum(rate(node_cpu_seconds_total{job="node_exporter"}[1m]))

    内存 #

    内存使用率 = （总内存 - 空闲内存 + 缓冲缓存 + 页面缓存） / 总内存 * 100

    (node_memory_MemTotal_bytes - node_memory_MemFree_bytes - node_memory_Buffers_bytes - node_memory_Cached_bytes ) / node_memory_MemTotal_bytes * 100

    node_memory_MemTotal_bytes：总内存大小 node_memory_MemFree_bytes：空闲内存大小 node_memory_Buffers_bytes：缓冲缓存大小 node_memory_Cached_bytes：页面缓存大小

    硬盘 #

    硬盘总容量

    sum (node_filesystem_size_bytes) by (instance)

    硬盘有效可用容量

    sum (node_filesystem_avail_bytes) by (instance)

    磁盘使用率

    (sum (node_filesystem_avail_bytes) by (instance) - sum (node_filesystem_avail_bytes) by (instance)) / sum (node_filesystem_size_bytes) by (instance)

    根目录总容量

    sum (node_filesystem_size_bytes{mountpoint="/"}) by (instance)

    根目录有效可用容量

    sum (node_filesystem_avail_bytes{mountpoint="/"}) by (instance)

    根目录使用率

    (sum (node_filesystem_avail_bytes{mountpoint="/"}) by (instance) - sum (node_filesystem_avail_bytes{mountpoint="/"}) by (instance)) / sum (node_filesystem_size_bytes{mountpoint="/"}) by (instance)

    网络 #

    网络上传速率

    rate(node_network_transmit_bytes_total{device="eth0"}[1m])

    网络下载速率

    rate(node_network_receive_bytes_total{device="eth0"}[1m])
